# Rendering — Sources

## Status

This document describes the target source architecture (vNext).

It fills the gap between:
- `engine.render.md` — current render-list-centric production/interchange model
- `engine.view.md` — views that reference renderable inputs
- `engine.technique.md` — techniques that consume typed inputs
- `engine.frame.recipe.md` — frame planning over those inputs

Current implementation: render lists only. This is a design target.

---

## Why Sources Exist

`Mel_Render_List` is already a useful engine primitive.
But it is too narrow to be the whole long-term data model.

Melody needs a render-side concept that can represent:
- CPU-fed sprite and text data
- retained ECS-backed scene data
- GPU-owned instance and meshlet databases
- render targets consumed by later techniques
- procedural/generated inputs

If the architecture keeps treating every input as "basically a render list",
it will eventually bend awkwardly around:
- modern GPU-driven pipelines
- postprocess and composition
- persistent GPU databases
- non-CPU-owned data

That broader concept is a **source**.

---

## Core Idea

A **source** is any persistent, typed render input or intermediate that a
technique can legally consume.

It is:
- not a view
- not a technique
- not a pass
- not a frame recipe

It is the thing that says:
- what kind of storage/ownership model this data uses
- what schema this data conforms to
- who is allowed to write it
- who is allowed to read it
- what its lifetime/update policy is

In short:

- **Source kind** = how data is stored and owned
- **Source schema** = what the data means
- **Source handle** = the actual instance

This is the right level for the engine to reason about render inputs without
collapsing everything into one storage type.

---

## Design Goals

1. **Render lists remain valid, but stop being the only model.**
   A render list should become one source kind, not the whole architecture.

2. **Modern GPU data must fit naturally.**
   GPU-generated instance databases, meshlets, draw packets, and visibility
   outputs must be first-class citizens.

3. **Techniques should consume explicit typed inputs.**
   A technique should say what kinds and schemas of sources it accepts.

4. **Views should attach sources, not storage assumptions.**
   A view references renderable inputs; it should not care whether those inputs
   happen to be a mapped CPU list or a GPU-only database.

5. **The planner/graph should understand ownership and access policy.**
   Source metadata must be rich enough for lifetime, synchronization, and
   compilation decisions.

---

## Source Model

The model should be built from three distinct concepts.

### Source Kind

How the data is stored and owned.

```c
typedef enum {
    MEL_SOURCE_LIST,
    MEL_SOURCE_RETAINED,
    MEL_SOURCE_GPU_BUFFER,
    MEL_SOURCE_TARGET,
    MEL_SOURCE_PROCEDURAL,
} Mel_Source_Kind;
```

### Source Schema

What the data means and how consumers interpret it.

Examples:
- sprite entries
- text runs
- mesh instances
- skinned mesh instances
- meshlets
- lights
- decals
- visibility buffer
- HDR color target

```c
typedef struct Mel_Source_Schema_Handle { u32 id; } Mel_Source_Schema_Handle;
```

### Source Handle

The actual source instance attached to views and consumed by techniques.

```c
typedef struct Mel_Source_Handle { u32 id; } Mel_Source_Handle;
```

This split is important.

Two sources can share one schema but use different kinds:
- a CPU-written sprite list
- a retained ECS sprite database

Same meaning. Different ownership/update model.

---

## Source Kinds

### `MEL_SOURCE_LIST`

Transient or CPU-fed structured entries.

Best for:
- sprites
- text
- debug primitives
- small or medium dynamic batches

This is basically today's `Mel_Render_List` model.

### `MEL_SOURCE_RETAINED`

Persistent structured data updated incrementally over time.

Best for:
- ECS-backed scene data
- retained UI geometry
- long-lived instance sets

This may be backed by the same storage machinery as a list, but it is a
different semantic kind.

### `MEL_SOURCE_GPU_BUFFER`

GPU-owned or GPU-primary structured data.

Best for:
- instance databases
- meshlet tables
- indirect draw packets
- culling outputs
- visibility/classification outputs

This is essential for modern rendering.

### `MEL_SOURCE_TARGET`

An image/target that becomes input to later techniques.

Best for:
- postprocess inputs
- shadow maps
- gbuffer attachments
- reflection captures
- resolved view outputs

This matters because not all technique inputs are structured records.

### `MEL_SOURCE_PROCEDURAL`

A source generated by logic rather than directly owned entry storage.

Best for:
- procedural terrain tiles
- synthetic particle streams
- generated debug geometry
- virtualized data feeds

This kind should stay available even if its initial implementation is thin.

## Access And Lifetime Policy

Every source should declare enough policy for the planner and graph to reason
about it.

Conceptually:

```c
typedef enum {
    MEL_SOURCE_ACCESS_CPU_WRITE = 1 << 0,
    MEL_SOURCE_ACCESS_GPU_WRITE = 1 << 1,
    MEL_SOURCE_ACCESS_CPU_READ  = 1 << 2,
    MEL_SOURCE_ACCESS_GPU_READ  = 1 << 3,
} Mel_Source_Access;

typedef enum {
    MEL_SOURCE_LIFETIME_FRAME,
    MEL_SOURCE_LIFETIME_RETAINED,
    MEL_SOURCE_LIFETIME_EXTERNAL,
} Mel_Source_Lifetime;
```

This lets the engine distinguish:
- ephemeral frame-built sprite lists
- retained ECS databases
- GPU-only persistent databases
- externally owned imported targets

That metadata should drive:
- graph barrier planning
- transient resource reuse
- planner validation
- debug tooling

---

## Suggested API Shape

Conceptually:

```c
typedef struct {
    str8 name;
    Mel_Source_Kind kind;
    Mel_Source_Schema_Handle schema;
    u32 access_flags;
    u32 lifetime;
    void* user;
} Mel_Source_Desc;

Mel_Source_Handle mel_source_create(const Mel_Source_Desc* desc);
void mel_source_destroy(Mel_Source_Handle source);
```

Convenience construction should exist for current engine primitives:

```c
Mel_Source_Handle mel_source_from_render_list(
    Mel_Render_List* list,
    Mel_Source_Schema_Handle schema);

Mel_Source_Handle mel_source_from_target(
    Mel_Render_Target* target,
    Mel_Source_Schema_Handle schema);
```

That lets the engine evolve without breaking the simple path.

## Render Lists Become A Source Kind

Current `Mel_Render_List` should remain a core primitive.

But architecturally it should be understood as:
- a very important source implementation
- especially good for CPU-driven production
- not the universal meaning of render input

In other words:

```text
render list < source
```

not:

```text
source == render list
```

That distinction becomes important the moment you want:
- GPU-owned visibility data
- render-target-as-input workflows
- non-CPU-generated geometry

---

## Relationship To Views

Views should attach sources.

Not:
- "sprite lists only"
- or "mesh lists only"

Those should remain convenience wrappers:

```c
void mel_view_attach_sprite_list(Mel_View_Handle view, Mel_Render_List* list);
void mel_view_attach_mesh_list(Mel_View_Handle view, Mel_Render_List* list);
```

But the real architectural seam should be:

```c
void mel_view_attach_source(Mel_View_Handle view, Mel_Source_Handle source);
```

This is what lets a view naturally work with:
- CPU sprite lists
- GPU meshlet databases
- offscreen render targets
- procedural sources

without changing what a view is.

---

## Relationship To Techniques

Techniques should declare the source kinds and schemas they accept.

Examples:

- sprite technique:
  accepts `LIST` or `RETAINED` with `SPRITE_SCHEMA`

- text technique:
  accepts `LIST` or `RETAINED` with `TEXT_SCHEMA`

- mesh shader technique:
  accepts `GPU_BUFFER` with `INSTANCE_SCHEMA`, `MESHLET_SCHEMA`,
  `MATERIAL_SCHEMA`

- postprocess technique:
  accepts `TARGET` with `HDR_COLOR_SCHEMA`

This is how techniques stop being fuzzy.

They do not merely consume "some data".
They consume explicit source contracts.

---

## Relationship To Frame Recipes

Frame recipes should plan over sources, not just views and techniques.

The planner needs to know:
- what inputs exist
- what kinds they are
- which techniques can legally consume them
- whether they are frame-local or retained
- whether they are CPU-fed or GPU-fed

That is how the planner makes coherent decisions about:
- technique resolution
- target reuse
- compilation validity
- fallback behavior

---

## Relationship To The Render Graph

At graph level, sources eventually become concrete resources:
- buffers
- images
- counters
- imported/external handles

But the graph should not be the place where source meaning is invented.

The source layer exists so the engine can say:
- what this input is
- how it is owned
- how it may be consumed

before compilation reaches raw pass/resource topology.

## Example: Simple 2D

```c
Mel_Source_Handle world_sprite_source =
    mel_source_from_render_list(&world_sprites, MEL_SCHEMA_SPRITE);

Mel_Source_Handle hud_text_source =
    mel_source_from_render_list(&hud_text, MEL_SCHEMA_TEXT);

mel_view_attach_source(world_view, world_sprite_source);
mel_view_attach_source(hud_view, hud_text_source);
```

The app still uses render lists.
The architecture now understands them as typed sources.

---

## Example: GPU-Driven Scene

```c
Mel_Source_Handle instance_source = mel_source_create(&(Mel_Source_Desc){
    .name = S8("instances"),
    .kind = MEL_SOURCE_GPU_BUFFER,
    .schema = MEL_SCHEMA_INSTANCE_DB,
    .access_flags = MEL_SOURCE_ACCESS_GPU_WRITE | MEL_SOURCE_ACCESS_GPU_READ,
    .lifetime = MEL_SOURCE_LIFETIME_RETAINED,
});

Mel_Source_Handle meshlet_source = mel_source_create(&(Mel_Source_Desc){
    .name = S8("meshlets"),
    .kind = MEL_SOURCE_GPU_BUFFER,
    .schema = MEL_SCHEMA_MESHLET_DB,
    .access_flags = MEL_SOURCE_ACCESS_GPU_READ,
    .lifetime = MEL_SOURCE_LIFETIME_RETAINED,
});

mel_view_attach_source(gameplay_view, instance_source);
mel_view_attach_source(gameplay_view, meshlet_source);
```

No fake render-list framing is needed.

## Schema Ownership

Schemas should likely be globally registered engine descriptors.

Examples:
- `MEL_SCHEMA_SPRITE`
- `MEL_SCHEMA_TEXT`
- `MEL_SCHEMA_MESH_INSTANCE`
- `MEL_SCHEMA_LIGHT`
- `MEL_SCHEMA_MESHLET_DB`
- `MEL_SCHEMA_HDR_COLOR`

Built-in schemas should be native engine constants or handles.
Custom schemas should be registrable by advanced users.

That gives techniques a stable way to describe what they consume.

---

## Inspection

Debug tooling should expose:
- source name
- source kind
- schema
- lifetime
- access policy
- owning system or module
- attached views
- consuming techniques

This will matter a lot once the engine supports mixed CPU/GPU-owned data paths.

---

## Open Questions

These need interface negotiation before implementation:

1. **Schema representation**
   Are schemas just IDs, or richer descriptors including stride/format/semantic
   metadata?

2. **List vs retained**
   Are these truly separate kinds, or one storage kind with different lifetime
   flags?

3. **Target-as-source**
   Does every render target automatically become a source, or is that an
   explicit wrapper step?

4. **Procedural source model**
   Is procedural a true source kind, or a producer that fills another source
   kind?

5. **Ownership**
   Are sources module-static objects, handle-registry objects, or wrappers over
   caller-owned storage?

6. **Import/export**
   How should external resources enter the source model?

---

## Summary

Sources are the missing data-model layer of Melody rendering.

They let the engine say:
- what render inputs exist
- how they are owned
- what they mean
- who can consume them

That is what lets the architecture scale cleanly from:
- CPU-written sprite lists
- to retained scene data
- to GPU-driven modern rendering

without pretending they are all the same thing.
