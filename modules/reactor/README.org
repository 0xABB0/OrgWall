#+TITLE: Reactor

Reactor is the main-loop abstraction for graphical applications.

A reactor drives a process's event pump and dispatches lifecycle
callbacks. The shape is borrowed from SDL3's "main with callbacks"
model, extended with multi-listener registration per lifecycle slot
and, on platforms that allow it, multi-instance reactors.

A graphical application that does not care about main-loop control
takes a =Mel_Reactor= handle and ignores it. An application that
cares uses the handle to register callbacks, post messages, or
spawn additional reactors.

* Lifecycle slots

Four slots accept listeners:

- =on_init= fires once before the loop starts,
- =on_event= fires for each posted message,
- =on_update= fires once per iteration, just before the loop sleeps,
- =on_destroy= fires once after the loop exits.

Registration order is firing order. Listeners that register after
=run()= has started do not retroactively receive =on_init=.

* Posting

=mel_reactor_post(reactor, message)= drops a =void*= into the
reactor's queue. The reactor delivers the message to every
=on_event= listener on its own thread, on the next iteration. The
caller owns the pointee until delivery.

Where the platform exposes a native thread-bound queue, the backend
uses it directly. Otherwise the backend keeps its own queue plus a
wake mechanism.

* Threading

A reactor binds to one OS thread for its lifetime. Listener
registration is safe from any thread before =run()= starts; once
the reactor is running, mutating operations must happen on the
reactor's own thread.

=mel_reactor_stop= and =mel_reactor_post= are safe to call from any
thread.

* Platforms

** macos
=CFRunLoop=. =on_update= is driven by a =CFRunLoopObserver= on
=kCFRunLoopBeforeWaiting=. =post= signals a =CFRunLoopSource= that
drains an internal inbox into =on_event= listeners.

** win32
Win32 thread message queue. =post= uses =PostThreadMessageW= with a
custom =WM_APP= code carrying the message pointer in =lParam=.
=stop= posts =WM_QUIT=. =on_update= fires once per drain cycle,
before =WaitMessage= sleeps.

** linux
=epoll= plus =eventfd= wake. Independent of any display server.
=post= pushes into an internal inbox and signals the eventfd; the
loop wakes, drains the inbox into =on_event= listeners, then fires
=on_update=.

** android, web, vr
Not yet implemented.

* Current limitations

- =mel_reactor_create= returns null on every platform: user
  reactors (additional loops on a second thread) are deferred.
- Scheduled work (timers) is not part of this module. It belongs
  in a separate module that consumes the reactor.
- Display-server integration (X11, Wayland, NSApp, HWND windowing)
  belongs in =gui.platform.*= modules.
