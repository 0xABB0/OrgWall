#+TITLE: GUI — TODO

Execution checklist for the GUI module. =readme.org= is the specification;
this file is what remains to do and the resume point for the next agent.

Status keys: TODO (not started), WIP (in progress), DONE.
Order within "First slice" is the intended build order.

* Architecture inversion — backend-owned API  [2026-05-23]

The portable "two-phase create" (=mel_<widget>_create → mel_gui__create →
mel_gui__backend_<widget>_create=) and the fat =Mel_Gui_Widget= are gone. The
public API now /is/ the platform implementation: =mel_<widget>_create_opt= and
the =mel_gui_*= ops are defined directly in =src/<backend>/=. The portable core
keeps only the slotmap registry, screen nav, layout, and handle accessors over a
thin =Mel_Gui_Node= (handle/parent/native/user/id + layout working set). See
readme.org "Architecture: backend-owned, no generic indirection".

Done across *all three* imperative backends (appkit, android, win32):
- Callbacks live on the native object (NSView subclass properties / a =jlong= on
  the Java listener / a per-=HWND= struct), wired only when an opt slot is set;
  the node carries none. =callbacks.c=, =mel_gui__fire_*=, =src/controls/= and
  =widget.c='s ops were deleted.
- Android ops are direct C→JNI on the cached =View= jobject (=w->native=); the
  =MelGui.views= map and its generic Java ops are gone, density is cached C-side.
  This closes the old "Per-control direct integration" and "Redundant w->native
  ref" items below.

Verification state:
- *appkit* — builds + runs (=./nob build app hello-world-gui=). Interactive
  click/slider/etc. not auto-tested in CI; eyeball recommended.
- *android* — =.so= (both ABIs) + Java + APK build clean
  (=./nob build app hello-world-gui android=). On-device run not yet exercised.
- *win32* — rewritten *blind*: this host has no win32 toolchain that accepts the
  codebase's clang-only features (=MEL_OVERLOADABLE=, C23 =nullptr=), so it was
  pattern-matched to the verified appkit/android shape and the prior win32
  structure. *Must be built with clang on Windows before trusting.*

Known gaps from the inversion (small, were latent before):
- =on_destroy= (lifecycle) no longer fires from the inverted backends — it used
  to come from the generic node->cb in =destroy_tree=. Re-wire per-backend from
  the native teardown if/when an app needs it. (Unused by hello-world-gui.)
- Runtime per-capability bulk setters (=mel_gui_set_*_cb=) were removed (they
  were unused and type-erasure-hostile under native-stored callbacks); add back
  per-backend when needed. Folds into the "Per-callback-slot setters" item.

   CheckBox, TextField, Slider, Canvas; =./nob build app hello-world-gui android=

   canvas =onDraw= bridges to =on_paint= (paint context passed through as an
   opaque =jobject= so the current hello-world-gui Windows-only =#ifdef=
   branch is unaffected). Reactor runs in =MEL_REACTOR_ATTACHED= mode on the
   Android main looper via the existing reactor android backend.
   Frame model: each =mel_frame= is a =FrameLayout= child of the activity
   root; =mel_app_present= calls =MelGui.presentFrame= which hides siblings
   and shows the requested one (mobile-class screen swap, not yet a real
   back stack — that's the "mobile/" class tier todo).
   JNI plumbing was consolidated: =JNI_OnLoad= moved to a new
   =modules/platform/src/android/jni.bridge.c=; midi now uses
   =mel_platform_android_env()=.
   Layout integration: =MelFrame.create= installs an =OnLayoutChangeListener=
   that fires =nativeFireResize(handle, w, h)= on size change (first layout
   pass, rotation), which calls =mel_gui__fire_resize= and re-runs the column
   layout. DPI is handled at the Java boundary: =MelGui.start= reads
   =DisplayMetrics.density=, =attach/setBounds= multiply incoming coords by
   density, =nativeFireResize= divides outgoing coords by density,
   =MelCanvasView= scales its Canvas and divides touch x/y. Net effect:
   every coordinate the C side sees is in density-independent units (dp on
   Android, matching macOS points).
   Architecture cleanup (2026-05-23 audit):
   - Java side split per-widget: =MelFrame=, =MelLabel=, =MelButton=,
     =MelCheckBox=, =MelTextField=, =MelSlider=, =MelCanvas= +
     =MelCanvasView=, =MelScreenFragment=, all under one slim =MelGui=
     core (start/stop, density, shared ops, presentFrame). Mirrors the
     per-widget structure of =modules/gui/src/macos/= and
     =modules/gui/src/android/=.
   - JNI registration is per-widget: each =modules/gui/src/android/<widget>.c=
     owns its =jclass= + =jmethodID= statics and exposes a register fn that
     =mel_gui__android_register= calls. =Mel_Gui_Android= struct shrank to
     shared cross-widget operations only. Adding a widget is now: new
     =<widget>.c= + =Mel<Widget>.java= + one line in
     =mel_gui__android_register=.
   - =w->native= now holds a =NewGlobalRef= jobject pointing at the Java
     View (mirroring macOS's =CFBridgingRetain= pattern). The old
     =(void*)1= sentinel is gone; =backend_destroy= releases the global ref.
   - Frame title moved from a Java =titles= =HashMap= to a fixed-size buffer
     in =Mel_Gui_Frame_Impl=. Java has no titles state; the C side is the
     source of truth and calls =setActivityTitle= when the frame is the
     visible one. =presentFrame= also takes the title as a parameter.
   - Navigation uses =FragmentManager= rather than a hand-rolled =ArrayDeque=:
     each Screen wraps in a =MelScreenFragment= and =mel_app_present=
     becomes =fm.beginTransaction().replace(...).addToBackStack(null).commit()=.
     =MelodyActivity= registers an =OnBackInvokedCallback= (Tiramisu+) and
     keeps =onBackPressed= for pre-33; both call =popBackStack= when count
     > 1 else =finish()=. The custom in-Activity navStack is gone.
   - Android =backend_set_focus= on a toplevel is now a no-op (was double-
     navigating because both =set_visible= and =set_focus= were calling
     =presentFrame=, pushing the same frame twice). Toplevel focus is
     implicit in being the currently-presented frame; multi-frame
     navigation goes through =set_visible= only.
   Open items before this can move to DONE:
   - Drop the deprecated framework =Fragment= navigation. =MelScreenFragment=,
     =MelGui.presentFrame= and =MelodyActivity.handleBack= use
     =android.app.Fragment= / =FragmentManager=, deprecated since API 28 and
     muted with =@SuppressWarnings("deprecation")=. The build pulls zero
     AndroidX (only =com.android.application=, no =dependencies {}= block), so
     the AndroidX =fragment= replacement is off the table by design. Fragments
     here are only a back-stack of top-level frames, and the backend already
     tracks every View by handle and owns the back gesture
     (=OnBackInvokedCallback= on Tiramisu+, =onBackPressed= pre-33). Replace
     them with a backend-owned stack of frame handles that swaps the
     container's single child View on present/back. Deletes =MelScreenFragment=
     and the =@SuppressWarnings=, removes the =FragmentManager= back-stack
     count logic from =handleBack=, and keeps the no-AndroidX stance. Same pass
     should retire the deprecated =WindowInsets.getSystemWindowInset*= /
     =consumeSystemWindowInsets= in =MelodyActivity.onCreate= (deprecated API
     30; matters under =targetSdk 36= edge-to-edge) for
     =getInsets(WindowInsets.Type.systemBars())= + =WindowInsets.CONSUMED=.
   - DONE Per-control direct integration, no generic indirection. See the
     "Architecture inversion" section at the top: listeners are installed only
     when the opt slot is set, callbacks live on the =View=, ops are direct
     C→JNI on =w->native=. Done for android + appkit + win32.
   - DONE Redundant =w->native= global ref. =w->native= is now the real, used
     =View= jobject (ops dereference it); the =MelGui.views= map is gone.
   - Modified-UTF-8 vs standard UTF-8 at the JNI text boundary.
     =mel_gui__android_jstring= uses =NewStringUTF= and the inbound paths
     (=backend_get_text=, =MelTextField.nativeTextChanged=) use
     =GetStringUTFChars=; both speak JNI *modified* UTF-8, not standard UTF-8.
     This is correct for ASCII and the whole BMP, and only mis-encodes
     supplementary characters (emoji, U+10000+, which become surrogate-pair
     6-byte sequences instead of 4). Fix all three paths together — outbound
     via =NewByteArray= + =SetByteArrayRegion= + =new String(bytes, "UTF-8")=,
     inbound via =String.getBytes("UTF-8")= → =byte[]= — caching the
     =java/lang/String= class, the =([BLjava/lang/String;)V= ctor and the
     =getBytes(Ljava/lang/String;)[B= method id once. Half-fixing one
     direction is incoherent, so it stayed deferred.
   - Drain UI events / wake from C: the reactor's android backend uses its
     own timerfd to schedule ticks; views are manipulated synchronously from
     C on the UI thread, which is correct, but =mel_reactor_post= from a
     non-UI thread should wake the looper — verify =ALooper_wake= path.
   - =keyDown:= forwarding for =EditText= — Android consumes most key events
     in the IME, so on_key_* on a textfield won't fire on Android today.
   - Per-backend escape hatches: =<gui/android/*.h>= class-tier headers
     exposing raw =android.view.View= access beyond the core API.
   - =savedInstanceState= / =Bundle= state preservation. Today if the OS
     kills the process under memory pressure, all C-side state is lost
     and the app restarts at the root screen. Real Android apps serialize
     the current screen + the user-visible bits of session state into a
     Bundle in =onSaveInstanceState= and restore in =onCreate=. Requires
     Melody apps to declare what state is screen-saveable.
   - Activity-result / intent system. No way today to launch a system
     activity (camera, share, file picker) and receive a result. The
     readme calls multi-window / native-menus first-class but doesn't
     yet name intent dispatch; the mobile class tier should.
   - Custom-Android-View extension story. An app that wants to ship a
     widget Android has no native equivalent for (e.g. a music keyboard)
     must today add code to =modules/gui/src/android/=. There is no
     out-of-tree extension point: a Melody app should be able to register
     its own =android.view.View= subclass as a widget kind without
     touching the gui module's android backend sources.
   - Opt-in multi-Activity =mel_app_present= mode. Today every Screen is
     a sibling Fragment-equivalent in one Activity. Some apps (deep
     linking, intent return, per-screen task isolation) want each Screen
     pushed as its own Android Activity. A =MEL_FRAME_OWN_ACTIVITY= flag
     on =Mel_Frame_Opt= would change =mel_app_present= for that frame to
     =startActivity=, while the default stays single-Activity (modern
     Android best practice). Gated on the mobile class tier.
   Lifecycle closed: =mel_gui_shutdown()= (public) walks the slotmap calling
   =mel_gui__backend_destroy= per widget (which decrements the frame
   counter and triggers =mel_reactor_quit= on the last toplevel), frees
   per-widget allocations, =mel_slotmap_free=s the widget store, calls
   =mel_gui__screens_reset=, and zeroes all module globals. =nativeStop=
   in =modules/app/src/android/app.bridge.c= calls it. =MelGui.stop=
   clears =views/activity/root/density= and tears down the FragmentManager
   container. After Back from the root screen finishes the Activity, the
   next =onCreate= re-runs =mel_app_setup= from a clean slate even when
   the process is retained.
   Status bar / nav bar inset closed: =MelodyActivity.onCreate= installs a
   =setOnApplyWindowInsetsListener= on root that pads by the system bar
   insets and consumes them, so frame children draw inside the safe area.
   - Class+methodID caching ships via =MelGui.nativeRegister()= called from
     =MelGui.start()=, because =FindClass()= from the reactor's deferred
     ALooper tick uses the system classloader (no Java→native transition in
     progress) and cannot resolve app classes. Documented here so the
     pattern is reused for any future backend that needs the app
     classloader from a non-JNI context.

* Deferred — subsystems

** WIP  Layout system
   First slice landed: =Mel_Layoutable= data struct embedded in every widget
   (=preferred/fixed_w_h=, =weight=, =cross_align=, four-side =margin=);
   =Mel_Layout= opaque + single sanctioned vtable (=measure=/=arrange=);
   =Mel_Align= = =DEFAULT/START/CENTER/END/STRETCH= (DEFAULT means "inherit
   container's cross_align"). =Mel_Column_Layout= is the one algorithm shipped:
   stacks children top-to-bottom, distributes leftover primary-axis space by
   =weight=, applies cross_align per child. Layouts are heap-allocated via
   =mel_column_layout(...)= and freed when the owning widget is destroyed.
   Trigger points: =mel_app_present= replaces bbox autosize with
   =measure+arrange= when the frame has a layout; =mel_gui__fire_resize=
   updates the widget's =width/height= and re-arranges if a layout is set.
   Default is still absolute positioning — set =.layout= on a parent to opt in.
   =hello-world-gui='s main screen demos the column; details stays on absolute
   coords to verify the no-layout path.
   Units: all coordinates the C side passes (in opts, in =Mel_Layoutable=,
   in pointer-event callbacks) are *logical* — density-independent. Backends
   translate to physical pixels at the boundary. macOS already works in
   points (1pt = 1dp). Android scales by =DisplayMetrics.density= in
   =MelGui.attach/setBounds= and inverts on =nativeFireResize= /
   =MelCanvasView=. Win32 currently maps 1dp = 1px (bare-by-default, no DPI
   awareness yet — see "Win32 opt-in polish APIs").
   Open items:
   - =Mel_Row_Layout= — horizontal sibling of column.
   - =Mel_Grid_Layout= — 2D with row/col spans + stretch factors per track.
   - Stack/Overlay layout — z-stacked siblings, naming still deferred.
   - Intrinsic-size query: today =preferred_h = 0= on a child means "use
     the widget's current width/height" which is whatever was passed in
     the opt. The right semantic is "ask the native control" — e.g. an
     =EditText= on Android needs ~40dp tall for its built-in padding,
     while an =NSTextField= on macOS is happy at ~22pt. Add a
     =mel_gui__backend_natural_size(w, *nw, *nh)= per-widget hook the
     layout consults when preferred is 0; ship per-widget defaults from
     each backend. Until then, opt heights are picked conservatively for
     the platform with the largest intrinsic.
   - Naming collision on =.layout= field: today every parent opt uses
     =.layout = Mel_Layout*= for the algorithm and every child opt uses
     =.layoutable = Mel_Layoutable= for the hints. Frame is the only first-
     slice parent so this works. When Panel/GroupBox land they need both
     fields on the same opt, which is already supported by the chosen naming.
