#+TITLE: GUI — TODO

Execution checklist for the GUI module. =readme.org= is the specification;
this file is what remains to do and the resume point for the next agent.

Status keys: TODO (not started), WIP (in progress). Completed work has been
removed; the shipped design lives in =readme.org=.

* Open items from shipped work

** Inversion lifecycle
- =on_destroy= no longer fires from the inverted backends (it used to come from
  the generic node->cb in =destroy_tree=). Re-wire per-backend from the native
  teardown if/when an app needs it. Unused by hello-world-gui today.

** Containers (Panel / GroupBox / ScrollView)
- ScrollView nested layout: a layout set on a ScrollView arranges children
  against the *viewport* height, not =content_h=, so layout-driven content does
  not fill the scrollable extent (the demo uses absolute child coords). A layout
  needs to measure/arrange against the content size when the parent is a
  scroll host.
- GroupBox/ScrollView focus is the only capability wired on the content node;
  pointer/keyboard on the container surface ship on Panel (appkit) only.

** Structural (Dialog / TabView / Splitter)
- TabView/Splitter page nodes are grandchildren of the frame, so the shallow
  one-level =mel_gui__destroy_tree= leaks their node slots (and, on appkit, the
  page host global ref) when the frame closes — a pre-existing limitation of the
  destroy walk that also affects nested Panel/GroupBox children.

** Insets / safe area
- *PAD width/height reduction in landscape.* Android =PAD= pads the Frame's
  =FrameLayout=, which honours it for child positioning, so portrait
  (status top / nav bottom) is correct. But the C column layout still arranges
  against the *full* Frame size, so a left/right inset (landscape nav bar, side
  notch) shifts children right without shrinking the arranged width — content can
  clip. Fix: feed =safe_area= into the layout as an outer margin (reduce
  available size + offset origin) rather than Java padding; folds into the WIP
  layout system.
- *appkit PAD child-offset.* For normal decorated windows =safeAreaInsets= is 0,
  so PAD is a visual no-op. Real child-offset only bites under
  full-size-content / custom-titlebar windows, which the desktop class tier
  introduces — wire PAD offset there.
- *Typed extras as escape hatches.* status-vs-navigation split,
  =tappableElement=, =waterfall= (Android); =keyboardLayoutGuide=-driven =ime=
  on appkit/uikit; web =env(safe-area-inset-*)= + =visualViewport= for =ime=.
- iOS/web/uikit inset fills land with those backends.

** win32 verification debt
Containers, Structural and Insets were written *blind* on win32 (no clang-on-
Windows host). Unverified until built with clang on Windows: the BS_GROUPBOX
inner-inset subclass, the WS_*SCROLL repositioning, the tab-control parent
subclass, the dialog owner-disable/result path, the custom splitter drag math,
and the (trivial) zero-inset path.

* Deferred — android follow-ups

Do not block the android backend; tracked for later.

- Modified-UTF-8 vs standard UTF-8 at the JNI text boundary.
  =mel_gui__android_jstring= uses =NewStringUTF= and the inbound paths
  (=backend_get_text=, =MelTextField.nativeTextChanged=) use
  =GetStringUTFChars=; both speak JNI *modified* UTF-8, not standard UTF-8.
  Correct for ASCII and the whole BMP, mis-encodes supplementary characters
  (emoji, U+10000+). Fix all three paths together — outbound via =NewByteArray=
  + =SetByteArrayRegion= + =new String(bytes, "UTF-8")=, inbound via
  =String.getBytes("UTF-8")= → =byte[]= — caching the =java/lang/String= class,
  the =([BLjava/lang/String;)V= ctor and the =getBytes(Ljava/lang/String;)[B=
  method id once. Half-fixing one direction is incoherent.
- Drain UI events / wake from C: the reactor's android backend uses its own
  timerfd to schedule ticks; views are manipulated synchronously from C on the
  UI thread, which is correct, but =mel_reactor_post= from a non-UI thread
  should wake the looper — verify =ALooper_wake= path.
- =keyDown:= forwarding for =EditText= — Android consumes most key events in the
  IME, so on_key_* on a textfield won't fire on Android today.
- Per-backend escape hatches: =<gui/android/*.h>= class-tier headers exposing
  raw =android.view.View= access beyond the core API.
- =savedInstanceState= / =Bundle= state preservation. Today if the OS kills the
  process under memory pressure, all C-side state is lost and the app restarts
  at the root screen. Real Android apps serialize the current screen + the
  user-visible bits of session state into a Bundle in =onSaveInstanceState= and
  restore in =onCreate=. Requires Melody apps to declare what state is
  screen-saveable.
- Activity-result / intent system. No way today to launch a system activity
  (camera, share, file picker) and receive a result. The readme calls
  multi-window / native-menus first-class but doesn't yet name intent dispatch;
  the mobile class tier should.
- Custom-Android-View extension story. An app that wants a widget Android has no
  native equivalent for (e.g. a music keyboard) must today add code to
  =modules/gui/src/android/=. There is no out-of-tree extension point: a Melody
  app should be able to register its own =android.view.View= subclass as a
  widget kind without touching the gui module's android backend sources.
- Opt-in multi-Activity =mel_app_present= mode. Today every Screen is a sibling
  in one Activity. Some apps (deep linking, intent return, per-screen task
  isolation) want each Screen pushed as its own Android Activity. A
  =MEL_FRAME_OWN_ACTIVITY= flag on =Mel_Frame_Opt= would switch
  =mel_app_present= for that frame to =startActivity=, default stays
  single-Activity. Gated on the mobile class tier.

* Deferred — subsystems

** WIP  Layout system
   Shipped: =Mel_Layoutable= per widget (=preferred/fixed_w_h=, =weight=,
   =cross_align=, four-side =margin=); =Mel_Layout= opaque + single vtable
   (=measure=/=arrange=); =Mel_Align=; =Mel_Column_Layout= (the one algorithm).
   Open items:
   - =Mel_Row_Layout= — horizontal sibling of column.
   - =Mel_Grid_Layout= — 2D with row/col spans + stretch factors per track.
   - Stack/Overlay layout — z-stacked siblings, naming still deferred.
   - Intrinsic-size query: today =preferred_h = 0= on a child means "use the
     widget's current width/height" (whatever the opt passed). The right
     semantic is "ask the native control" — e.g. an =EditText= on Android needs
     ~40dp tall, an =NSTextField= ~22pt. Add a
     =mel_gui__backend_natural_size(w, *nw, *nh)= per-widget hook the layout
     consults when preferred is 0; ship per-widget defaults from each backend.
     Until then, opt heights are picked conservatively.

** WIP  Unified Canvas drawing API
   Shipped: =Mel_Painter= immediate-mode surface handed to Canvas =on_paint=.
   Ops: =clear=, =fill_rect=, =fill_ellipse=, =stroke_rect=, =draw_line=,
   =fill_round_rect=, =draw_text=. Geometry reuses =math.geo=/=math.vector=;
   =Mel_Color= is RGBA8.
   Open items / deferred surface:
   - Arbitrary paths (move/line/curve/close, fill + stroke), polygons.
   - Images / bitmaps (=draw_image=), with source + dest rect and scaling.
   - Clip regions (rect + path) and affine transforms (translate/scale/rotate).
   - Gradients (linear/radial) and per-shape alpha/blend modes.
   - Text: only a single run with the system font + size ships; rich text,
     font family selection, measurement (=measure_text=), alignment and
     wrapping are deferred. Android/win32 text goes through JNI modified-UTF-8 /
     UTF-16 — same supplementary-char caveat as the other text paths.
   - Drawn backend: implement =mel_painter_*= by recording into the shared
     draw-list its CPU/GPU renderers play back.
